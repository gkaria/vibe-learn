description: Summarize recent session activity or answer a specific question about what was built, grounded in the session log

Read .vibe-learn/session-log.jsonl to understand what has happened in this session.

To pass a question or topic, type it as a follow-up message after invoking `/prompts:learn`.

---

## Mode: `obsidian:recall <topic>`

If the user types `obsidian:recall` followed by a topic:

Extract the topic from the user's message. If no topic is given, ask the user what they want to recall.

1. Load the Obsidian config: check `.vibe-learn/obsidian.json` first, then `~/.vibe-learn/obsidian.json`. If neither exists, run the config setup flow (ask vault path, subfolder, global vs project save location) before proceeding — same flow as the `obsidian` mode below.
2. Construct the search path: `<vault_path>/<subfolder>/`
3. Search that directory for markdown files matching the topic — search across filenames, headings (`#`), body text, and frontmatter tags using the Grep tool.
4. Read up to 5 of the most relevant matching files.
5. Synthesize a summary covering:
   - Which sessions touched this topic and when (use the `date` frontmatter field)
   - Key decisions and patterns related to the topic
   - Any unchecked `- [ ]` items in "Things to Study" sections on this topic
   - Connections or progressions across sessions (e.g., "You first set this up on March 2nd, then extended it on April 5th")
6. Present the summary to the user. **Do not write any file.**

## Mode: `obsidian` or `obsidian <question>`

If the user types `obsidian` (optionally followed by a question):

Extract any question text after `obsidian` (may be empty).

**Step 1 — Load config:**
Check `.vibe-learn/obsidian.json`, then `~/.vibe-learn/obsidian.json`.

If no config is found:

- Ask the user: "What is the path to your Obsidian vault? (e.g. /Users/me/MyVault)"
- Ask: "Which subfolder should vibe-learn notes go in? (default: Development/Sessions)"
- Ask: "Save this config globally for all projects (`~/.vibe-learn/obsidian.json`) or just for this project (`.vibe-learn/obsidian.json`)? [global/project, default: global]"
- Write the config to the chosen location (creating the directory with `mkdir -p` if needed):

  ```json
  {
    "vault_path": "<their answer>",
    "subfolder": "<their answer or default>",
    "tags": ["vibe-learn"],
    "link_style": "wikilink",
    "include_project_tag": true,
    "note_naming": "{date}-{project}"
  }
  ```

  Global path: `~/.vibe-learn/obsidian.json`
  Project path: `.vibe-learn/obsidian.json`

**Step 2 — Generate content:**

If no question was given: summarize the most recent session activity in plain language — what was just built or changed, decisions made, patterns used.

If a question was given: answer that specific question grounded in the session log and relevant source files.

**Step 3 — Write the note:**

- Determine the project name from the `cwd` (last path component of `.vibe-learn/../` or the working directory).
- Determine the date (today's date, YYYY-MM-DD).
- Compute the filename using the `note_naming` template from config (default: `{date}-{project}`), appending `-learn.md`. On filename collision, append a numeric suffix (e.g., `-2.md`).
- Target path: `<vault_path>/<subfolder>/<filename>`
- Create the subfolder if it doesn't exist (using Bash: `mkdir -p`).
- Write the note with this format:

```markdown
---
date: <YYYY-MM-DD>
created: <ISO 8601 timestamp>
project: <project-name>
tags:
  - vibe-learn
  - <project-name if include_project_tag is true>
type: learn
---

# Learn Note — <YYYY-MM-DD>

<generated content>

---
*Generated by [vibe-learn](https://github.com/gkaria/vibe-learn)*
```

Confirm to the user: "Saved to `<full path>`."

## Mode: plain learn (no qualifier)

If invoked with no follow-up (or a plain question):

If no question: Summarise the most recent actions in plain language — what was just built or changed, the decisions behind it, and any interesting patterns or concepts used. Format as a readable list starting with "📘 **What just happened:**". Keep it tight — 3-5 points max.

If the user provides a question: Answer that specific question. Ground your answer in the session log and read relevant source files with your Read tool. Explain as if talking to someone learning to code — clear, no jargon, specific to what actually happened in this session.

In both cases: if the session log is empty or only has prompts with no tool events, say so and offer general help instead.
